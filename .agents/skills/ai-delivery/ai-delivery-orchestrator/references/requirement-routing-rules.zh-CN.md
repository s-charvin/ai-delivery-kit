# 需求路由规则

当默认入口流程需要推荐以下任一操作时，使用这些规则：

- `continue req-xxx`（继续 req-xxx）
- `create req-yyy`（创建 req-yyy）

始终只产生一个推荐，然后在采取任一路径之前暂停等待人工确认。

## 证据来源

在推荐路由之前，检查当前仓库状态：

- 现有的 `.ai-delivery/requirements/*` 包
- 需求标题和摘要
- `todo.md`、`status.json` 和 `traceability.json`
- 新用户输入中的源材料链接、文件名、标识符和功能范围

## 仅在以下情况推荐 `continue req-xxx`

仅当以下所有条件成立时，才推荐继续现有需求：

1. 候选需求仍然活跃。
   - 它尚未完全合并、归档或处于其他终结状态。
2. 新材料明确指向相同的产品范围。
   - 示例：相同的功能、相同的页面/流程、相同的 API 范围、相同的需求文档，或可追溯性中已出现相同的受管控源链接。
3. 没有强有力的冲突证据表明这是一个不同的交付流。

当推荐 `continue` 时，引述至少两个具体的证据条目。

## 在以下情况推荐 `create req-yyy`

当以下任一条件成立时，推荐创建新需求：

- 没有活跃的需求具有足够的匹配证据
- 最接近的候选需求已合并、归档或处于其他终结状态
- 新材料引入了意义不同的功能范围、验收目标或交付流
- 继续现有需求会将不相关的产品真实数据混入同一个受管控包中

如果证据混合或较弱，优先选择 `create req-yyy`。

## 人工门控

在推荐 `continue` 或 `create` 之后：

1. 展示唯一的推荐路由
2. 展示其背后的证据条目
3. 暂停等待人工确认或覆盖

如果人工覆盖了推荐，遵循人工决策并从此处继续受管控的工作流。
