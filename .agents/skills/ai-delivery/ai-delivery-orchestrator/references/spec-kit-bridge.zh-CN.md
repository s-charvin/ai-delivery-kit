# Spec Kit 桥梁

- 目的：保持官方 `speckit-specify`、`speckit-plan` 和 `speckit-tasks` 在上游，同时为它们提供切片范围的受管控上下文。
- `prepare-speckit-context` 读取：
  - `slice-contract.md`
  - `interaction-design.md`
  - `traceability.json`
  - 对 UI 切片读取 `ui-acceptance-contract.yaml`
  - 当 API 影响存在时读取 `api-contract-mapping.md`
- 它写入一个精简的上下文包：
  - `spec-kit-input.md`
- `spec-kit-input.md` 是一个传输产物，而非真实数据。真实数据仍在 `.ai-delivery` 中。
- 该包应携带：
  - `requirement_id`
  - `subreq_id`
  - `slice_id`
  - `slice_type`
  - 源产物路径
  - 反向可追溯性引用
  - 从受管控门控推导出的规划约束
- 官方 `speckit-*` 应消费 `spec-kit-input.md` 以及标准的 `.specify` 模板，无需仓库本地的技能本体重写。
- 如果在下一个 Spec Kit 步骤之前源真实数据发生变更，首先重新生成 `spec-kit-input.md`。
- 示例产物：[contacts-friends-idle/spec-kit-input.md](examples/contacts-friends-idle/spec-kit-input.md)
