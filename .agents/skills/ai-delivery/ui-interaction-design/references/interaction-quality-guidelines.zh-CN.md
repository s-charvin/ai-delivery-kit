# 交互质量指南

当 `ui-interaction-design` 需要在微交互、加载、反馈、动效、时序和可访问性方面选择更好的有边界默认值，且不改变业务含义时，使用此参考。

此指南特意设计为项目本地化。代理应直接使用此仓库参考，不应假设已安装任何外部交互设计技能。

## 有目的的动效

动效应仅在能改善理解时使用。

有效目的：

- 反馈：确认操作或状态变化已发生
- 方向感：显示内容来自何处或去向何方
- 焦点：吸引用户关注重要的更新或错误
- 连续性：在状态或表面变化时保持上下文

避免仅为了装饰、娱乐或让 UI 感觉更"高级"而不帮助用户理解的动效。

## 反馈表面阶梯

选择能保持清晰度的最轻量反馈表面：

1. 内联反馈
2. 字段或组件本地反馈
3. 区块或页面本地反馈
4. 短暂的全局反馈，如提示消息或 snackbar
5. 阻塞或中断性反馈，如模态框或全页面状态

指导：

- 对于验证和可恢复的错误，优先选择内联或本地反馈
- 当用户可保持在当前上下文时，使用 toast 或 snackbar 进行非阻塞状态确认
- 不要仅依赖 toast 处理关键、持久或表单特定的错误
- 仅在源真相或平台规则要求明确确认时使用阻塞性中断

## 加载模式阶梯

选择与用户操作匹配的最窄加载范围：

1. 控件级加载
2. 内联区块加载
3. 容器或面板加载
4. 页面级加载
5. 应用级阻塞加载

指导：

- 对于单操作提交，优先选择按钮或控件级加载
- 当保留布局有助于方向感时，优先选择内联进度或骨架屏
- 仅在布局已知且足够稳定以保留用户上下文时使用骨架屏
- 避免全屏或阻塞加载，除非用户确实无法安全继续
- 说明加载期间编辑是否可能保持
- 说明重试是本地、全局还是不可用

## 动效模式指导

当允许动效时，优先选择在 Web、iOS 和 Android 上保持可读性的模式：

- 状态强调：微妙的淡入淡出、缩放或高度变化以确认状态
- 容器连续性：当内容展开、折叠或切换时保持父子关系
- 进入和退出：使表面的到达和消失可理解
- 进度连续性：指示工作正在进行中而不隐藏上下文
- 焦点恢复：在验证或错误后保持用户注意力在变化区域附近

避免：

- 针对常规操作的大规模编排转场
- 阻止中断或恢复的动效
- 掩盖内容、焦点或错误可见性的动效
- 纯悬停效果且无触摸或键盘等效实现

## 时序指导

使用时序作为指导，而非硬性要求，除非源已定义。

建议范围：

- 微反馈：`80-150ms`
- 小状态变化：`150-250ms`
- 组件或面板转场：`200-350ms`
- 主要连续性转场：`300-500ms`
- 任何超过 `500ms` 的：说明为什么需要

指导：

- 短反馈应感觉即时
- 较长的转场应传达有意义的连续性
- 重复交互不应累积明显的延迟
- 如果时序不是基于源的，将其记录为有边界的假设

## 输入模态和可访问性

交互质量必须在指针、触摸、键盘和辅助使用中保持一致性。

将可访问性视为交互契约本身的一部分，而非可选的清理工作。

### 键盘和焦点

- 交互控件应保持键盘可达
- 焦点应在验证、加载和错误转场中保持可见
- 错误后，将焦点保持或恢复到恢复操作最清晰的位置
- 短暂反馈后，避免意外的焦点跳转

### 触摸和指针

- 不要仅依赖悬停作为唯一提示
- 如果悬停具有意义，提供等效的焦点或触摸可见状态
- 当手势快捷键影响重要操作时，应有可见或可发现的非手势替代方案

### 减少动效

- 当源真相未说明时，默认尊重减少动效偏好
- 即使在动效最小化时也保持状态清晰度
- 不要仅通过动画隐藏成功、错误或进度的含义

### 公告和状态语义

- 重要的加载、成功和错误状态应保持语义清晰
- 持久性验证或阻塞性错误不应仅以视觉方式传达
- 禁用状态应保持可理解，尤其是在它们阻止完成时

## 常见交互质量模式

### 验证

- 优先选择与字段和风险级别匹配的验证时序
- 对明显的格式问题使用即时验证且不会产生噪音
- 当持续中断会妨碍输入流程时，使用失焦或提交时验证

### 成功反馈

- 当用户保持在相同上下文时，本地确认通常足够
- 仅当变化是破坏性的、跨表面的或容易被忽略时使用更强的确认

### 错误恢复

- 错误状态应说明用户下一步可以做什么
- 当源真相允许时，优先选择可恢复的上下文内重试
- 区分短暂的传输错误和持久的业务规则错误

### 手势

- 仅当手势已经由源或现有模式建立时，才使用手势驱动的交互
- 关键操作需要可见的回退路径
- 手势完成阈值应感觉有意为之，而非一触即发

## 常见故障模式

- 过度动画增加了愉悦感但隐藏了含义
- 本地进度已足够时仍使用阻塞性加载
- 仅通过 toast 显示关键错误
- 纯悬停的可发现性
- 验证或模态返回后焦点不可见
- 成功反馈在用户感知之前消失
- 没有可见回退的手势
- 忽略减少动效期望的动效
- 基于源的流程尚未允许时，乐观反馈即暗示成功
